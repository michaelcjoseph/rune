- [ ] Jarvis MCP server times out on complex `kb_query` requests, and serves `/mcp` from the same process + event loop as the webview — a heavy query degrades both. _(moved from ideas.md, 2026-06-22)_
  - **Issue** (two distinct roots — latency and isolation)
    - **Latency: `kb_query` runs a full multi-step agent in-request, unbounded.** The `kb_query` handler (`src/mcp/server.ts:55-67`) awaits `queryKB(question)`, which spawns the `kb-query` Claude Code subprocess via `runAgent('kb-query', prompt, undefined, …)` (`src/kb/query.ts:52`). That agent is agentic, not a single completion — its prompt tells it to read `knowledge/index.md`, open multiple wiki pages, grep the vault, then synthesize (`query.ts:42-50`). A complex question is tens of seconds to minutes of multi-turn work.
    - **No request timeout, no streaming, no cache.** `runAgent`'s third arg is `timeoutMs`, and the call passes `undefined` (`query.ts:52`), so the run is unbounded server-side. The MCP transport has only a 30-minute idle-*session* timer (`SESSION_IDLE_MS`, `src/server/mcp-transport.ts:61`), never a per-*request* deadline. Nothing on the server caps the call, so the MCP *client* (Claude App / Claude Code) hits its own request timeout first and reports failure while the agent keeps running. There's no progress/streaming to keep the client alive and no result cache, so repeating a question pays the full cost again.
    - **Isolation: `/mcp` shares the webview's HTTP server, event loop, and process.** `startHttpServer` builds ONE `http.createServer` and dispatches health → oauth → `mcpHandler` → `webviewHandler` in a single request callback (`src/server/http.ts:112-125`). MCP and the webview are the same Node process on the same single-threaded event loop. The query path also does synchronous blocking vault I/O *before* the agent spawns: `searchWithFilter` + `searchVault` read files with `readFileSync` (`src/kb/search.ts:101`) on the request thread (`query.ts:26-33`), stalling the loop — including any in-flight webview request — for the duration of the scan. The spawned agent subprocess then competes with the daemon for CPU, the Claude CLI, and file handles. A heavy `kb_query` degrades webview responsiveness, exactly the coupling this bug names.
  - **Fix options** (A/B target latency, C/D target isolation; independent, both wanted)
    - A. (latency — bounded + resumable) Convert `kb_query` from an in-request agent run to an async job: return fast with a job id, run synthesis in the background, let the client poll or stream progress. At minimum, pass a real `timeoutMs` to `runAgent` (`query.ts:52`) so the server fails fast with a clear error instead of running unbounded after the client has already given up.
    - B. (latency — cheaper synthesis) The full `kb-query` agent is overkill for most questions. Tier it: serve simple lookups from `searchWithFilter` + a single completion over the top-K pages (no multi-turn agent), and reserve the full agentic run for genuinely complex questions. Cache answers keyed by normalized question + KB content hash.
    - C. (isolation — get MCP off the webview's loop) Move the `/mcp` mount and the KB synthesis to their own process/port, or run synthesis in a worker, so the webview event loop is never blocked by KB work. The shared `http.Server` (`http.ts:104-133`) is convenient but couples two very different workloads — sub-ms UI requests vs minute-long agent runs.
    - D. (isolation — stop blocking the loop synchronously) Even before A/C, make the pre-agent vault search async/off-thread (or precompute a persistent index) so `searchVault`'s `readFileSync` scan (`search.ts:101`) can't freeze in-flight webview requests on every query.
