- [ ] `/work --auto` run wedges open after the agent finishes — terminal `result` event never triggers a process-tree reap, so orphaned test runners strand the run as `running` for hours.
  - **Issue**
    - The runner keys run-completion entirely on `child.on('close')` (`src/jobs/work-runner.ts:717`) — the `while (!done …)` generator loop (`:743`) only ends when the child's stdio streams all close. It does NOT key off the agent's terminal `result` event in the stream; that event flows through as an ordinary `output` line.
    - The child is spawned with `stdio: ['ignore', 'pipe', 'pipe']` (`:300`) and no `detached`. Any grandchild the agent spawns (e.g. a `vitest run` via a Bash tool call) inherits those stdout/stderr fds. If a grandchild hangs on open handles, the parent's pipes never close, `close` never fires, `done` stays false, and the generator parks indefinitely even though the agent already emitted `result`.
    - The only teardown path is `ctx.cancel()` → `child.kill('SIGTERM')` (`:746`), which signals the direct child only — not the process group. With no `detached: true` + `process.kill(-pid)`, orphaned grandchildren survive a cancel anyway.
    - Supervision detects the silence but can't fix it: the quiet-run nudge (`src/intent/supervision.ts`, project 11 Phase 4) fires at most once (`quietNudgedAt`, `:130`) and `isStalled` keys on `lastChildAliveAt` — a child-alive-but-silent run is "quiet," never "stalled," so it's never force-terminated. The run sits `running` until a human kills it.
    - Confirmed instance: run `19cd198f` (project 09-expand-cockpit, 2026-06-03). Agent emitted `result: success` at 20:48 CT after 25 commits (Phases 1–3 shipped, Phase 4 tests written), then the tree stayed alive ~3.5h. Four orphaned `vitest run` PIDs (spawned 18:14 / 19:13 / 19:35 / 20:25) held stdio open; the quiet nudge fired once at 21:07, then the run sat `running` with a no-output alert every minute until manually killed — only then did supervision flip it to `failed`.
  - **Fix options** (A is the real fix; C is the safety net)
    - A. Drive teardown off the terminal `result` event, not stdio close. When the stream yields the agent's terminal result, mark `done`, capture exit facts, and actively reap the tree (SIGTERM → grace → SIGKILL) instead of waiting on `close`.
    - B. Spawn with `detached: true` and kill the whole process group (`process.kill(-child.pid, …)`) on both teardown and cancel, so orphaned grandchildren (hung vitest) are always reaped — `:746` currently signals only the direct child.
    - C. Escalate supervision past the one-shot nudge: if a run is quiet (or a `result` was seen) and hasn't closed within N seconds, force-kill the tree and let classification proceed from the captured result — a watchdog mirroring `planning-expiry`, so the cockpit self-heals without a human.
    - D. (defense-in-depth) Prevent the leak at the source: wrap test invocations in the run with a hard timeout / guaranteed-exit flags so a single hung `vitest` can't wedge the whole run.
