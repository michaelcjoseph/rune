# Subsystem Deep-Dives

Architecture mechanics that don't change what Claude does in a typical session but matter when working *in* a given subsystem. `CLAUDE.md` carries a one-line summary of each; the detail lives here. Per-file annotations are in `module-reference.md`.

## Provider-aware chat sessions

Telegram, cockpit Home, and product chat all enter through `askChatWithContext()` (`src/ai/chat.ts`). It resolves the session's selected alias through `policies/model-policy.json` and delegates to `askClaudeWithContext` or `runCodex`; the transport handler does not choose a provider itself. `DEFAULT_CHAT_MODEL` and `CONVERSATION_MODEL` are `gpt-5.6-terra`, while Claude-only one-shots, agents, and session summarization retain Claude aliases.

Each persisted `Session` keeps Rune's public `sessionId` plus optional executor metadata. Claude bindings store `{format:'claude', sessionId?}`; Codex bindings record `authority` (`read-only` or `product-full-access`), the effective cwd, and the scoped writable root. New sessions begin unbound. Restored records that predate executor metadata migrate to Claude using the legacy Rune session id. Explicit legacy Codex `writeEnabled:true/false` records normalize to the corresponding authority (legacy read-only records also normalize their omitted cwd to the vault); records with no posture metadata are unknown and never resume. Same-provider turns resume a Codex thread only when it has a non-empty thread id and its authority, effective cwd, and scoped root exactly match the current routing decision. A missing thread id, provider, privilege, repo, or scoped-root change starts a fresh correctly sandboxed thread and bootstraps it from the stored transcript capped at 40,000 characters. This carries visible conversation context across boundaries without treating one provider's private session/tool state or a stale Codex authority as portable. `/fresh` resumes Claude-backed sessions for summarization but summarizes Codex-backed sessions from the stored transcript.

The transport handler derives one provider-neutral authority from scope plus workspace resolution: a resolved product is `product-full-access`; Home/global and unresolved products are `read-only`. That single value selects the system-prompt capability, tool set, product MCP registration, Codex sandbox, and persisted binding. Claude full-access product chats retain repo cwd, the writable-root hint, development tools, and `--dangerously-skip-permissions`.

Codex persistence is opt-in at the primitive: chat passes `persistentSession:true`, captures `thread.started`, then calls `codex exec resume` with the stored thread id. Resumes inherit the thread's original sandbox and therefore never receive a conflicting `-s`; any unknown, weaker, or differently-bound thread is rotated before dispatch and its replacement receives transcript replay. All pre-existing `runCodex` callers remain `--ephemeral` by default. Both providers register chat operations with the shared in-flight registry, so `/cancel`, cockpit status, shutdown tracking, and path scrubbing retain the same behavior. Global and unresolved-product Codex chat are read-only; fresh full-access product threads request `danger-full-access`, while every Codex chat retains the secret-scrubbed child environment.

Full-access configured product chats also build a provider-equivalent MCP registration through `product-chat-mcp.ts`. The shared `isolated-mcp-config.ts` encoder produces Claude's complete strict inline registration and Codex's replacement `mcp_servers` table from one server/tool/timeout definition; artifact-role MCP uses the same encoder. User config is ignored, the server is required, and the same seven tools are pinned and approved on both fresh and resumed turns. The child environment is assembled from an explicit safe allowlist plus server-owned `RUNE_PRODUCT_CHAT_PRODUCT`; it does not load `.env.local`, inherit Rune/integration secrets, or receive dummy Telegram credentials. `config.ts` keeps the Telegram fields lazy for this narrow import graph, while `src/index.ts` calls `assertOperatorConfig()` before the full operator process boots. Product-scoped `kb_query` synthesis explicitly invokes `runAgent(..., userVisible=false)`, so it neither reads the operator's Telegram id nor emits operator-only bus/in-flight frames. Home/global and unresolved-product turns retain their existing MCP posture and never receive the Cockpit diagnostics.

The transport handler wraps the complete conversation transaction in `withChatTurnLock`: key = transport+user for global chat, product+transport+user for product chat. Session read/create, provider call, transcript append, and executor persistence therefore cannot race for one scoped conversation, while unrelated sessions remain concurrent; a failed turn releases the queue normally. When a provider switch, `/fresh`/`/clear`, or a failed first Codex turn makes a thread unreachable, `cleanupCodexThread` validates the id and best-effort removes only its matching rollout files under `$CODEX_HOME/sessions`.

The Home selector includes GPT-5.6 Terra and synchronizes to the active webview session model; with no session, Terra is selected. Product chat exposes `/gpt-5.6-terra` alongside the Claude quick commands. Existing sessions retain their persisted model rather than being silently moved to the new default.

## In-flight op tracking

Model children register an `InFlightOp` (`src/transport/in-flight.ts`) and emit `BusOpEvent` frames (start/progress/end). TG shows a tracker message ("­¤żö agent ┬Ę 12s ┬Ę /cancel") that edits every ~10s and deletes on end; the webview shows a cancellable pill. Classifier ops are filtered from senders to avoid resolver spam.

Cancellation uses the provider-neutral `OperationCancellation {operationId, source, requestedAt}` record; source is exactly `telegram`, `cockpit`, or `internal`. `/cancel [opId-prefix]` and `POST /api/ops/:id/cancel` stamp their explicit transport source, and internal callers must pass `internal` explicitly. `cancelOp` is synchronous: the first accepted request is stored before SIGTERM, later requests cannot overwrite its source or timestamp, and the record stays on the live op until the executor captures it immediately before unregistering. Claude and Codex capture on both child `close` and `error`, so an error-event race cannot erase attribution.

## Mutation pipeline

`src/transport/mutations.ts` is the central registry for autonomous codebase operations (`MutationDescriptor`, applier registry, `createMutation`/`cancelMutation`). The appliers, all registered in `src/index.ts`:

1. **`workRunApplier`** (`src/jobs/work-runner.ts`) ŌĆö spawns Claude CLI with `spec.md + tasks.md + /work --auto` for a project slug.
2. **`genEvalLoopApplier`** (`src/jobs/gen-eval-loop-runner.ts`, `autoApprove: false`) ŌĆö validates the `gen-eval-loop` payload and drives `runGenEvalLoop()` (per-round: createWorktree ŌåÆ /work --auto ŌåÆ /review ŌåÆ recordRound ŌåÆ evaluateLoop ŌåÆ destroyWorktree).
3. **`orchestratedWorkApplier`** (`src/jobs/orchestrated-work-runner.ts`, `autoApprove: true`, kind `orchestrated-work`) ŌĆö creates a sandboxed worktree, drives `runProjectOrchestration` over real fs/git effects with `createProductionTaskWorkflowRunner` as the live per-task role-spawn binding (Phase 8), maps the terminal `OrchestrationResult` to a single `MutationEvent` (finalizedŌåÆcompleted, heldŌåÆcompleted-flagged-held, blockedŌåÆfailed).
4. **`workRunReleaseApplier`** (`src/jobs/work-run-release.ts`, `autoApprove: true`, kind `work-run-release`, `supervised: false`) ŌĆö drives a cold finalize-or-discard of a sentinel/manual-check parked run.
5. **`workRunAnswerApplier`** (`src/jobs/work-run-answer.ts`, `autoApprove: true`, kind `work-run-answer`, `supervised: false`) ŌĆö answers a parked `AskUserQuestion` and resumes the original parked worktree.
6. **`writingRunApplier`** (`src/jobs/writing-run-runner.ts`, `autoApprove: true`, kind `writing`) ŌĆö /blog and /writing-critique: drives the writing pipeline (research via in-process MCP tool handlers ŌåÆ writer-SOUL one-shot model stages ŌåÆ privacy-sanitized artifact ŌåÆ commit) against the `writing` product's repo on a create-or-resume `rune-writing/{slug}` branch; V1 publish is branch-commit only (no deploy). Pipeline states surface as run-feed log lines (`publishDerivedRunEvent` gate includes `writing`); a 30s keep-alive ticker shields the silent one-shot model calls from the kind-agnostic quietŌåÆcancel backstop; the worktree is destroyed at terminal on success AND failure (committed work lives on the branch; a kept dirty tree would wedge the slug); one run per slug at a time.

The cockpit Start action dispatches to either the orchestrated or legacy applier via the `src/jobs/work-dispatch.ts` seam ŌĆö `resolveWorkDispatch` maps the `resolveDispatchMode` decision to a concrete mutation kind; `handleApiMutationsCreate` substitutes `kind:'orchestrated-work'` when orchestrated mode is selected, stamps `dispatchMode`+`fallbackReason` onto the payload, and always records which applier ran. Both `orchestrated-work` and `work-run` share the same deterministic per-project worktree path and the same per-project + global concurrency caps, so the two never run the same project concurrently.

Mutations are logged append-only to `logs/mutations.jsonl`; orphaned `running` entries are flipped to `failed` at startup via `reconcileOrphans()`.

**Shutdown-park suppression:** `setMutationShutdownInProgress(v)`/`isMutationShutdownInProgress()` is armed by `shutdown()` (`src/index.ts`) before `killActiveProcesses()`. While armed, `startApply` suppresses ALL terminal/crash persistence and heartbeat upserts, but ONLY for `orchestrated-work` mutations ŌĆö a SIGTERM'd child would otherwise surface as a terminal the run never earned, flipping a boot-resumable `running` mutation to `failed` or clobbering a just-written shutdown park. Other mutation kinds keep their existing crash semantics. A suppressed run's `activeRuns` handle is also retained (not deleted in the `startApply` finally) so the parker ŌĆö which snapshots `activeRuns` only after the children are dead ŌĆö can still find the run after its applier unwinds. See the shutdown-park entry below for what owns the suppressed run's on-disk state instead.

### Supervision coupling

Every mutation state transition drives `src/jobs/supervision-store.ts`: `createMutation` seeds a `SupervisedRun` (`running` for autoApprove, `blocked-on-human` for pending-approval); `startApply` flips to `running`; `output` events refresh `lastHeartbeatAt` and advance `lastOutputAt` (both throttled to once per 30s ŌĆö `lastOutputAt` is the LLM-output signal the quiet-run nudge keys on, distinct from `lastChildAliveAt`); verified activity uses `recordRunActivity`, whose atomic read-modify-write deliberately clears `quietNudgedAt` so a later quiet period can form a fresh cycle; non-output writes thread the current `lastOutputAt` through so it is never reset to undefined; `completed`/`failed` and applier-crash paths flip terminal status. A superseded max-runtime request also stamps optional `maxRuntimeEpochAt`; legacy rows omit it and continue measuring from `startedAt`. `runTargetFromDescriptor` resolves and persists the same `{kind:'project'|'bug',slug}` target on supervision rows and run-feed events, with legacy payload fallbacks. Routine supervision writes are fail-safe; cancellation supersession fails closed if its required activity-reset write cannot persist. The audit source of truth remains `mutations.jsonl`.

## Supervision & stall-check

`SupervisedRun[]` current state persists to `logs/supervised-runs.json` (`supervision-store.ts`). `upsertRun` **field-merges** by id so a keep-alive heartbeat can't accidentally clear `quietNudgedAt`; `recordRunActivity` is the explicit clearing operation when actual output proves the quiet period ended. The store then applies a 768 KiB visibility budget by preserving every running/parked row and retaining only the newest terminal rows that fit. Terminal history remains durable in per-run summaries and `mutations.jsonl`; this keeps the diagnostics complete-reader below its 1 MiB fail-closed ceiling during normal operation. If active rows alone exceed the cap, they are never dropped and diagnostics correctly remain unavailable until state contracts. The stall-check runner (`stall-check-runner.ts`, 30s tick) runs on a single snapshot:
- `checkStalledRuns` ŌåÆ newly-nudged-id set (child-dead stall, 5min threshold).
- `planQuietNudges` over the non-stalled remainder (quiet-but-alive, keyed on `lastOutputAt`, 5min threshold) ŌåÆ publishes `formatQuietNudge`, persists `quietNudgedAt`.
- **Escalation (project 15 P2.7):** `planQuietCancel` (quiet past `WORK_RUN_QUIET_CANCEL_AFTER_MS` since nudge ŌåÆ `cancelMutation(..., 'system', 'quiet-run')`), `planMaxRuntimeKills` (running past `WORK_RUN_MAX_RUNTIME_MS` since `maxRuntimeEpochAt ?? startedAt`, regardless of liveness ŌåÆ source `max-runtime`, fail-toward-kill on corrupt timestamp).
- `planParkedNudges` over `blocked-on-human` runs (one-time staleness nudge after `PARKED_RUN_NUDGE_AFTER_MS`, never auto-releases).

Per-run send/persist/cancel/kill errors are individually caught so a failure on one run can't skip the rest; tick exceptions caught so a failure can't crash the server.

## Work-run lifecycle

`workRunApplier` spawns Claude CLI with `--output-format stream-json --verbose` and converts each stdout envelope into human-readable `output` `MutationEvent`s via the stream-jsonŌåÆdisplay adapter (`work-run-transcript.ts`). The terminal sequence:

- Tees raw envelopes to a per-run durable transcript sink (`<id>/transcript.jsonl`).
- Classifies on work product via `computeWorkProduct` + `finalizeWorkRun` (`work-run-classify.ts`) ŌĆö outcome Ōłł branch-complete / partial / noop / dirty-uncommitted / failed.
- Flushes transcript (awaits finish), writes `summary.json` atomically with the durable project-or-bug target identity and optional scrubbed nested-role cancellation correlation, appends a `WorkRunIndexRow` to `logs/work-runs/index.jsonl` (best-effort).
- Augments the terminal event with `projectSlug`+`product` so the cockpit and `formatWorkRunTerminal` can name the run.

**Start event (project 13 Phase 1a):** once `createWorktree` succeeds, yields a one-shot `start` `MutationEvent` carrying `operatorWorktreePath` (un-scrubbed) + `runId`/`projectSlug`/`product`. **The path is LOCAL-OPERATOR-ONLY and is never copied onto the descriptor**, so it never reaches `mutations.jsonl`/summary/index/transcript/forensics. The createWorktree-failure branch returns before this point so a run with no worktree emits no `start`.

**Parked state (project 13 Phase 1b + AskUserQuestion loop):** when the stream contains either a valid `RUNE_WORK_RUN_SENTINEL` (parsed via `parseWorkRunSentinel`) or a Claude `AskUserQuestion` assistant `tool_use` (parsed via `work-run-question.ts`), and the run was NOT user-cancelled, work-runner writes a durable `blocked-on-human` supervision record FIRST, SKIPS `runFinalizer` (leaves the worktree live), and yields a terminal event carrying `parked:true` + un-scrubbed `operatorWorktreePath`. Sentinel/manual-check parks carry `pendingCheck`/`command`/`reason` and render a one-tap "Release" action (`work-run-release:<id>`) that cold-finalizes or discards. Question parks additionally persist `SupervisedRun.parkedQuestion` (`source:'ask-user-question'`, question, stable option ids, labels/values/descriptions, `toolUseId`, `askedAt`) and render one answer action per option (`work-run-answer:<runId>:<optionId>`). A malformed `AskUserQuestion` still parks with an inspect-transcript fallback, never silently classifies as `noop`/`partial`.

**Answer resume:** `requestWorkRunAnswer(runId, optionId)` validates the parked run, structured question, option id, and live worktree, then creates the auto-approved `work-run-answer` control mutation. The applier flips the original run id back to `running`, spawns Claude in the existing worktree with the selected answer and retained worktree-derived `RUNE_VITEST_CACHE_DIR` injected, appends to the original transcript, and yields terminal bus events keyed to the original run id. A second `AskUserQuestion` parks again with a fresh `parkedQuestion`; otherwise the existing gated finalizer path classifies/lands/holds the resumed worktree. This path is distinct from Release: answering continues work, Release finalizes a non-question park.

**Commit-poll ticker (project 11 Phase 4):** `streamProcess` runs a parent-side commit-poll ticker (10s, unref'd, re-entrancy-guarded) ŌĆö each tick calls `git log baseSha..branch`, reads `tasks.md` tally via `parseTasks`, passes both to `planCommitProgress`, and enqueues a scrubbed throttled `progress` `MutationEvent` when a new commit lands. Enabled only when `sandbox.baseSha` is present.

**Forensics (project 11 Phase 3):** `exportForensics` writes a reconstructable evidence bundle to `logs/work-runs/<id>/` BEFORE the worktree is destroyed; best-effort by contract.

**Model-facing diagnostics:** `work-run-diagnostics.ts` projects summaries, supervision, recent task records, and display transcript tails for the dedicated product-chat MCP surface. Authorization is server-owned and fail-closed across every persisted ownership record before task/transcript readers run; unknown and foreign runs share one unavailable response. Summary reads are tri-state: a genuinely missing summary may fall back to complete supervision evidence, while invalid, malformed, or unreadable summary evidence denies access. `readAllRunsBounded` reads the supervision snapshot only when the complete file fits its byte cap and returns `complete:false` for oversized, unreadable, malformed, or partly invalid state; diagnostics refuse authorization whenever completeness is false, so omitted older ownership evidence cannot widen access. DTOs expose the persisted project-or-bug target identity (legacy records fall back to project) and the durable `{role,operationId,source,requestedAt}` cancellation record after the live child is gone. Index/task JSONL and transcript display readers share the fixed-byte text-tail primitive in `jsonl-tail.ts`, which reports source truncation and drops a partial leading line. Readers also bound record counts, transcript bytes (512 KiB), strings, nested object entries, and final JSON (64,000 characters), while recursively redacting secrets and scrubbing host pathsŌĆöincluding cancellation fields. They never return raw JSONL, artifact/worktree paths, cookies, or mutation controls.

**Isolated Vitest caches:** every active worktree maps to an opaque deterministic cache at `<os-temp>/rune-vitest-cache/<sha256(absolute-worktree-path)>`. Rune forces that path through `RUNE_VITEST_CACHE_DIR` for sandboxed agents, both initial and answer-resumed legacy work-run Claude children, and closeout/gate validation commands; inherited or product-credential values cannot override it. `vitest.config.cjs` consumes the variable and, by being CommonJS, avoids Vite's separate ESM config-bundle write to shared `node_modules/.vite-temp`. A parked worktree keeps its cache for resume. Normal/idempotent destruction, preserved-worktree reclaim, reconciliation rollback, integration-gate teardown, and orphan cleanup remove the matching cache best-effort; cleanup failures never mask the worktree result.

**Retention GC:** `gcWorkRuns` prunes `logs/work-runs/` artifacts oldest-first to satisfy `WORK_RUN_RETENTION_MAX_RUNS` + `WORK_RUN_RETENTION_MAX_BYTES`; protected set = activeRuns + non-terminal run-store + worktree-checked-out branches. Runs at startup and on each run completion (fire-and-forget).

## Gated-merge finalizer (project 15)

`runFinalizer(input, effects)` (`work-run-finalizer.ts`) is the shared idempotent phase-recorded finalizer, records a durable phase per step (`logs/work-runs/<id>/phase`) for crash-resume.

- **`hold` mode** (P0.4a): classify ŌåÆ flush transcript ŌåÆ write summary/index ŌåÆ resolve worktree (best-effort removal, branch left intact) ŌåÆ terminal supervision write. NEVER merges/pushes/deletes; never leaves the run `running`.
- **`gated-merge` mode** (P1.5 + Phase 3.5, live via `runGatedMerge`): classify ŌåÆ flush ŌåÆ summary ŌåÆ index ŌåÆ gate ŌåÆ merge ŌåÆ push ŌåÆ delete ŌåÆ terminal, recording `merged-not-pushed` then `pushed-not-deleted` (**push-before-delete: origin is the durable backup**); consults `readLastPhase()` for crash-resume (skips an already-committed merge/push/index-append ŌĆö exactly-once); a failed gate STOPS at branch-complete and alerts (never merges); a non-branch-complete run never consults the gate.

The pure gate DECISION (`evaluateGate`), gate RUNTIME (`runGate` ŌĆö runs validationCommands in an integration worktree so a red check never mutates `main`, with the integration worktree's isolated Vitest cache forced into each command and removed in teardown), and per-product/per-base-branch merge lock (`withBaseBranchLock`) are all wired as effects from `work-runner.ts`. `resolveWorktreeAndFinalize` runs `onBranchDelete` AFTER `removeWorktree` and BEFORE `git branch -d` (git refuses to delete a checked-out branch).

Live consumers: the live work-runner `apply()` terminal path (gated-merge mode) and the startup recovery path (`recovery-finalize-runner.ts` ŌĆö hold mode by default, gated-merge to RESUME a crashed-mid-merge run off its durable phase, never to initiate one).

## Orchestrated work & product-team agents (project 14)

The orchestrated path is OFF by default (`ORCHESTRATED_WORK_ENABLED=false`); a per-product `orchestratedMode: true/false` in `policies/products.json` overrides it. When on, the cockpit Start action routes to `orchestratedWorkApplier`.

**The six product-team roles** (`src/roles/loader.ts`): `pm`, `tech-lead`, `qa`, `coder`, `reviewer`, `designer`. Each has a SOUL charter and a `memory.md` under `agents/<role>/` (mirroring `agents/writer/`). Two-channel authority boundary: SOUL.md ŌåÆ system-prompt, memory.md ŌåÆ low-authority reference fence. Judgment roles run on anthropic via the Claude CLI (pm, tech-lead ŌåÆ fable; reviewer, designer ŌåÆ opus); artifact roles run on openai via the Codex CLI (qa ŌåÆ gpt-5.6-terra; coder ŌåÆ gpt-5.6-sol). Reviewer/coder provider distinctness is enforced fail-closed at resolution time in `team-task-deps.ts`. Aliases are policy data ŌĆö `policies/model-policy.json` is the source of truth, not this list.

**Artifact-role vault reads:** a product may opt explicit QA/coder sessions into `artifactMcp: "rune-kb-readonly"` (`writing` and `brand` do). Eligibility is checked from `ExecutionAgentOpts.role` at the execution-agent boundary, so tech-lead test-repair sessions and every other role receive no artifact MCP even on an opted-in product. Products without the policy gain no new Codex registration and retain prior Claude behavior.

`buildArtifactMcpConfig` is a pre-spawn, fail-closed, macOS-only boundary: platforms without Seatbelt are rejected. It validates the absolute Node/Seatbelt executable, TypeScript loader, broker and relay entrypoints, a readable/searchable vault directory, and the writable worktree. After generating the profile it runs a real Seatbelt preflight that starts `/bin/sh` in the worktree and creates/removes a probe file there. Policy/configuration, filesystem, profile, shell-startup, worktree-write, broker-startup, or index-readiness failure returns scrubbed task evidence beginning `rune-kb not registered:` before a model process starts, so no role round is consumed. The privileged `artifact-readonly.ts` broker receives the vault path once over stdinŌĆönot argv or environmentŌĆöstarts with an empty spawn environment, scrubs its environment again before Rune imports, builds the index in memory without `initKB()` or scaffolding writes, and constrains followed symlinks to targets inside the vault's real root. It then serves exactly `vault_search`, `journal_range`, and `follow_wikilinks` over a temporary Unix-domain socket. The model registration launches only `artifact-readonly-relay.ts`, whose argv contains the socket path and whose environment is empty; neither process fabricates Telegram credentials, and the relay knows no vault path, Rune/integration secret, daemon OAuth setting, or localhost endpoint.

Both Claude and Codex model children are wrapped once in `/usr/bin/sandbox-exec` with one generated mode-0600 Seatbelt profile. File writes default-deny and are reopened only for the lexical/real worktree, the invocation's private runtime/temp directory, and required device paths; configured/real vault paths remain denied for direct reads and writes, and the lexical/real profile file is hidden from the child. The profile also denies reads of the operator's source Codex/Claude/SSH/cloud/config homes and common secret literals such as `.env.local`, denies raw local TCP access to `localhost:*`, and permits only the broker Unix socket for local vault transport. Both providers receive only `TMPDIR` as provider-neutral runtime environment. Codex, and only Codex, additionally receives a private per-invocation `HOME`/`CODEX_HOME` under that same runtime directory, seeded with only the operator's `auth.json`, so CLI authentication and state/log/database writes stay inside the outer profile without exposing the rest of the operator Codex config. Codex model-generated shell commands are pinned with `shell_environment_policy.inherit="none"` so they do not inherit that auth/runtime environment. Claude receives no Codex auth environment and uses `--strict-mcp-config` with only the relay registration. Codex uses the mutually-exclusive `RunCodexOpts.externallySandboxed` mode, which emits `--dangerously-bypass-approvals-and-sandbox`, requires a profile path, and omits `-s workspace-write`, preventing a nested Seatbelt application while the outer profile remains authoritative. It also uses `--ignore-user-config` and an argv-safe replacement of the complete `mcp_servers` table: `rune-kb` is `required=true`, `enabled_tools` is exactly the three approved reads, `default_tools_approval_mode="approve"` permits their non-interactive use, and startup/tool timeouts are 10s/60s. The three tools advertise `readOnlyHint:true`, `destructiveHint:false`, and `openWorldHint:false`. The detached broker is registered with Rune's active-process supervision. Every eligible QA/coder invocation creates its own registration and broker; its idempotent stop path SIGTERMs then SIGKILLs the process group, unregisters it, and removes the runtime directory after that invocation. Broker signals or parent IPC disconnect destroy every accepted socket before closing, so a held relay cannot block shutdown.

**Flow:** `runProjectOrchestration` (`project-orchestrator.ts`) drives tasks to completion: while an unchecked task remains: `selectNextTask` ŌåÆ `assembleTaskContext` ŌåÆ `runTaskWorkflow` (the workflow owns per-task convergence internally) ŌåÆ cancellation boundary ŌåÆ on ready-for-closeout: `performCloseout` (curate+`applyContextUpdate` ŌåÆ `markSelectedTaskComplete` ŌåÆ runCloseoutChecks ŌåÆ commitCloseout ŌåÆ verifyCleanWorktree). Cancellation state carries source through `ApplyContext` ŌåÆ runner ŌåÆ `OrchestrationDeps`. User, shutdown, recovery, and unspecified system requests stop before closeout. Quiet/max-runtime requests also stop blocked/failed/cancelled evidence, but `ready-for-closeout` is strong enough to atomically clear the still-active watchdog request; the orchestrator emits one scrubbed `system-cancel-superseded` output (durable transcript + live run feed), then closes out and selects the next task. `runCloseoutChecks` uses the product's optional `closeoutValidationStrategy` under the short `WORK_RUN_CLOSEOUT_COMMAND_TIMEOUT_MS` budget: absent/`product-commands` runs the configured command list, while `vitest-related` argv-spawns Vitest related tests for normalized, deduplicated tracked changes against `HEAD` plus untracked files (deleted paths excluded); Rune and Rune-MCP opt in. A red check enters a bounded coder repair loop (`CLOSEOUT_REPAIR_CAP` = 2 whole-workflow re-runs with the scrubbed failing output tail as qaŌåÆcoder `GateRejectionFeedback`); the durable failure file retains bounded head + tail. On timeout, the validation runtime sends `SIGUSR2` before reap and writes sanitized Node reports (environment removed) plus command/head/tail evidence under `<run>/validation-diagnostics/`; every live/recovery gate supplies that per-run directory. Exhaustion best-effort WIP-commits the worktree (`commitWip`) and parks `blocked-on-human` (branch + worktree preserved, releasable via the standard release path ŌĆö a held terminal would be invisible to release and its git-registered worktree would block the project's next Start). Other closeout failures hold durably ŌåÆ advance. When no unchecked task remains: `buildFinalizerHandoff` ŌåÆ `runFinalizerHandoff`; its hard merge gate still runs the complete product `validationCommands` with `WORK_RUN_GATE_COMMAND_TIMEOUT_MS`. **Never self-merges** ŌĆö a completed orchestrated run holds branch-complete for operator merge (spec req 17). `context.md` is Rune-owned orchestration state; `context-curator.ts` is its only writer.

**Validation isolation and closeout selection:** product-controlled validation receives only basic toolchain environment plus its worktree-derived cache; Rune/integration secrets are absent. On macOS, Seatbelt denies external networking but permits localhost for test servers. Timeout reports use `validation-report-bootstrap.cjs`: report flags reach npm/npx's direct runner, then are removed before Next/Vitest worker creation. A timeout does not resolve until SIGTERMŌåÆSIGKILL process-group reaping completes. For `vitest-related` closeout, filenames beginning `-` are prefixed with `./` because the subcommand has no `--` terminator; deletions and global package/lock/compiler/test/build/loader configuration changes instead run the product's full validation commands.

**Restart resumability (durable cursor + salvage):** closeout persists a fail-closed checkpoint (`persistRunCheckpoint` ŌĆö a write failure blocks the run, since it persists the completion facts recovery reconstructs from) carrying a resumable cursor (`writeRunCursor`) with `currentTaskId: null`. The orchestrator ALSO writes a task-start cursor right after task selection, before the workflow runs (`persistTaskStartCursorSafely`, best-effort ŌĆö `commitWipSafely` precedent: a failed write must not hold a run that has done no work): `buildRunCursor` takes an optional `currentTask` arg whose non-null `currentTaskId` marks the named task in-progress/untrusted. Effect: a restart at ANY point after the first task selection is now findable by boot recovery ŌĆö previously the cursor only existed after the first closeout, so a mid-first-task restart was orphaned (or parked by the shutdown parker below). Recovery still re-derives position from tasks.md + records regardless (`reconstructRun` handles empty records: `{completedTaskIds: [], nextTask: first unchecked, drift: false}`), so the interrupted task is re-selected and re-run from scratch through the full role workflow ŌĆö a half-finished task is never trusted, only inspectable. **Restart salvage:** recovery redispatch (`orchestrated-work-runner.ts`'s apply-loop recovery branch ŌĆö the funnel for both boot recovery and cockpit `POST /api/work-runs/:id/recover`) reuses the worktree exactly as the interrupted run left it; after the sandbox is rebuilt from `recovery.worktreePath` and before `runOrchestration` re-runs, a dirty tree is best-effort WIP-committed (`rune(<product>): WIP ŌĆö restart salvage ŌĆö <project>`, via `commitWorktreeWip` ŌĆö generalized from the shutdown parker's `commitShutdownWip`, now shared by both) so the re-run task starts from a clean tree and closeout's `git add -A` can't silently absorb the interrupted task's dirt; the half-work stays inspectable on the branch instead.

**Shutdown park:** `shutdown()` (`src/index.ts`) arms the mutation-pipeline suppression above, then after children are dead calls `parkInFlightOrchestratedRuns()` (`orchestrated-work-runner.ts`) for each active running orchestrated run with source `shutdown` (irrevocable); if a resumable cursor exists, leave it `running` so boot recovery auto-resumes it as today (parking would degrade a routine restart to a human-gated release); if there is no cursor (mid-first-task ŌĆö previously orphaned and its diff discarded), resolve the deterministic branch/worktree, skip if the worktree is missing, best-effort WIP-commit a dirty tree, then write a `completed`+`parked:true` terminal + blocked-on-human supervision row ŌĆö visible and releasable in approvals after restart via the standard release path. Operator-triggered active-run recovery likewise tags the old invocation's cancellation source `recovery`, so verified task output cannot supersede the ownership handoff. The applier's own `finally` skips `destroyWorktree` while shutdown is in progress, and `persistTerminalMutationState` no-ops, so nothing races the parker's writes. Composes with boot recovery: resumable runs still auto-resume next boot; only non-resumable runs are parked. **With task-start cursors now written at every task selection, this parker is the rare fallback** ŌĆö a task-start cursor covers virtually every in-flight run, so parking now only fires on a task-start-cursor write failure or a restart in the narrow pre-first-selection window; most graceful restarts auto-resume instead of parking.

**Single task workflow** (`team-task-workflow.ts`): one task through role gates in order ŌĆö reviewer-independence check (fail-closed) ŌåÆ QA-first (tests or no-code-test rationale) ŌåÆ tech-lead test review (on first rejection the tech-lead **repairs** the tests itself instead of bouncing to the same QA blind spot: worktree patch guarded to `*.test.ts(x)` paths only via a pre-repair `git write-tree` snapshot delta with on-disk revert of violations, confirm-red against the product `validationCommands` ŌĆö green/timeout rolls the patch back ŌĆö then re-review; `repairable:false` or any repair failure degrades to the QA bounce Ōēż cap) ŌåÆ bounded coderŌåÆreviewer round loop (the coder drives the complete product `validationCommands` green in its worktree before handback ŌĆö full-suite self-gate, prompt-enforced; this is independent of the later closeout strategy) ŌåÆ objection-class hard gate ŌåÆ tech-lead diff review (reviewer + tech-lead sanction any test deleted/weakened in the diff against the coder's `TEST-REMOVED` handoff-note justifications) ŌåÆ designer IFF `task.designerNeeded` ŌåÆ PM wrap-up ŌåÆ `TaskEvidence`.

Cockpit poll interval widened from 1s to 5s (the file-backed sections only change once a minute; a 1s poll bought nothing).

**MCP daemon launchd runbook:** the committed LaunchAgent is `launchd/com.jarvis.rune-mcp.plist`, installed as `~/Library/LaunchAgents/com.jarvis.rune-mcp.plist` by `scripts/install-rune-mcp-launchd.sh`. It runs with `Label=com.jarvis.rune-mcp`, `WorkingDirectory=/Users/jarvis/workspace/rune`, and `ProgramArguments=/usr/bin/env npm run mcp:start`; the package script expands to `NODE_ENV=production node --env-file-if-exists=.env.local --import ./scripts/register-ts.mjs src/mcp/daemon.ts`, so daemon env comes from that working directory's `.env.local` when present. Required live-use expectation: set `RUNE_MCP_SECRET`; set `RUNE_MCP_ISSUER_URL` to the final public Funnel HTTPS origin; optionally override `RUNE_MCP_OAUTH_STORE_FILE`, `RUNE_MCP_HOST`, and `RUNE_MCP_PORT` (defaults: `logs/rune-mcp-oauth-store.json`, `127.0.0.1`, `3848`). The plist sets only `PATH`; it does not inline secrets. Stdout/stderr go to `~/Library/Logs/rune/rune-mcp.out.log` and `~/Library/Logs/rune/rune-mcp.err.log`; the daemon `/health` response also points at the in-repo process log.

```bash
scripts/install-rune-mcp-launchd.sh lint
scripts/install-rune-mcp-launchd.sh install
scripts/install-rune-mcp-launchd.sh restart
scripts/install-rune-mcp-launchd.sh uninstall

launchctl print gui/$(id -u)/com.jarvis.rune-mcp
launchctl kickstart -k gui/$(id -u)/com.jarvis.rune-mcp
launchctl bootout gui/$(id -u)/com.jarvis.rune-mcp
```

The install script creates `~/Library/LaunchAgents` and `~/Library/Logs/rune`, lints the source and installed plist, bootouts any existing `com.jarvis.rune-mcp` job, bootstraps the installed plist into `gui/$(id -u)`, then kickstarts it. `restart` is a kickstart only; `uninstall` bootouts the job and removes the installed plist.

Tailscale Funnel is the public Claude App ingress. Keep the daemon bound to loopback and map the Funnel HTTPS listener to the daemon:

```bash
tailscale funnel --bg --https=443 http://127.0.0.1:3848
tailscale funnel status
```

The resulting `https://<machine>.<tailnet>.ts.net` hostname is a human provisioning prerequisite, not a build artifact. Put that exact origin, with no trailing path, in `RUNE_MCP_ISSUER_URL`; Claude App then reaches `https://<machine>.<tailnet>.ts.net/mcp` while the daemon still listens locally on `127.0.0.1:3848`.

## Cockpit view

`buildCockpitView(registry, runStatus, taskProgress?, workRuns?, backlogCounts?, dispatchModes?)` (`src/intent/cockpit.ts`) is a pure projection. `CockpitProject` carries `lifecycleStatus`, `runStatus`, `actions`, and optional `progress` (gen-eval-loop block), `taskProgress` (done/total), `workRun` (id, outcome, reason, last-N output, startedAt, transcriptUrl), `dispatchMode`+`fallbackReason`. `CockpitProduct` carries optional `backlogCounts` (product-name-keyed, repo-backed only). The cockpit filters out lifecycle-`done` projects per product (keeping every product header). `handleApiCockpit` (`webview.ts`) feeds live `RunStatusByProject` from `readCockpitRunStatus(SUPERVISED_RUNS_FILE)` + `readWorkRunProjections` (passing the supervision store's running/blocked-on-human runs as the 4th `activeRuns` arg so a live run's card renders immediately).

**Webview UX:** the cockpit sidebar polls `GET /api/cockpit` and renders products/projects with lifecycle status, run-status, per-project action buttons (start / continue / enter-planning-mode), the in-flight gen-eval-loop progress block, a static task-progress bar, and a per-project work-run block (`renderWorkRun` ŌĆö active run shows elapsed + last-N output inline; terminated run shows typed outcome + reason + transcript link). The backlog drawer (`#backlog-drawer`) opens from a product's count line, fetches `GET /api/backlog/:product`, renders Bugs/Ideas tabs each with a Plan button + a `+` add chip. The Pending Approvals panel polls `GET /api/approvals`. A production-only "Ōå╗ Restart server" button posts to `POST /api/server/restart`.
